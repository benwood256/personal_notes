** M-x 

This allows you to run Emacs commands only (doesn't speak to the terminal)
Commands are series of words separated by hyphens (i.e. org-indent-mode)

*Everything in Emacs is a Lisp function*

This means that every keybinding is mapped to a function call, but there are tens of thousands of functions
So, certain very useful ones are remapped
Example -> C-x C-s = M-x save-buffer
or         C-x C-f = M-x find-file

There's no difference between the above shortcuts or running M-x <function name>

Evil has it's own keymaps in this stack

So, when the keybindings 'ciw' is pressed, Emacs finds the keymap for this in the Evil section
& this maps to a Lisp function within Evil (Change Inside Word)

** init.el & Variables, Functions and Hooks

3 things make up an init.el file: Variables, Functions & Hooks


Variables = what Emacs should remember
They control how things behave by storing a setting
Examples: telling Org to hide * / _, set indentation

Functions = what Emacs can do
They do something when called
This is all the built in Lisp functions in Emacs, you can just turn some on permaenently by placing them in the init.el file
Examples: line numbers 

Hooks = when Emacs should do something
A hook is a variable that contains a list of functions to run when a particular event occurs
They run functions automatically when something happens
i.e. when Org mode starts, enable visual-line-mode (when X happens, run this)


Variables = config
function = custom behaviour
hooks = when it runs

** Sources of packages

There are 3 main sources for packages:

1. Built in (internal) packages

   These ship with Emacs
   Examples: org-mode, flyspell, dired, calendar, project, tramp
   These aren't installed, you just need to enable them

   File here are located: /usr/share/emacs/29.3/lisp


2. MELPA packages (technically still an external package)

   MELPA is an Emacs specific package repository (i.e. an app store)
   The function: use-package is specific to MELPA packages

   Files here are located: ~/.config/emacs/elpa


3. External packages

   GNU ELPA = official packages
   MELPA = community packages

   These are not part of Emacs, but are supported by Emacs
   Example: Evil, Abyss theme (these are MELPA packages)
   You install it by -> package-install RET evil RET

   Files here are located: ~/.config/emacs/elpa (almost always but can't be guaranteed)

   Other examples of package managers supported by Emacs: GNU

** How packages work

*Package Manager*

   _package.el_ is the standard Emacs package manager
   Downloads packages from archives (MELPA, GNU ELPA, Non-GNU ELPA)
   Installs them to ~/.config/emacs/elpa
   Handles versions & dependencies

   There are other alternatives to package.el

*Step by step process*

*** You request a package

   M-x package-install RET evil RET
   i.e. find a package called evil in my configured archives & install it

*** Emacs checks the local database

   Emacs looks at ~/.config/emacs/elpa/archives
   This contains cached package lists: available version, dependencies, metadata, etc...

*** (Optional) Refresh the package list

   Emacs might run: package-refresh-contents
   This contents to MELPA or GNU ELPA to look for updates

*** Resolve package + dependencies

   Emacs then: finds evil, check dependencies & builds an install plan

*** Download package files

   Emacs downloads the actual package, from MELPA/GNU server
   Usually a .tar archive (i.e. evil-2.15.0.tar)

*** Install into local directory

   Emacs extracts it (i.e. ~/.config/elpa/evil-2.15.0/)
   Now this is physically installed

*** Add to load-path automatically

   Emacs updates its internal search path (i.e. load-path += ~/.config/elpa/evil-2.15.0/
   This tells Emacs: "you can now find Evil when someone asks for it"
   
*** Autoload setup

   _package.el_ creates autoload definitions so Emacs doesn't load the whole package immediately
   This means, Emacs knows the function exists & it only loads it when needed/requested

*** You manually activate the package

   In the config file you run:

   - (require 'evil)
    Emacs searches its load-path
    It finds evil.el inside ~/.config/emacs/elpa/evil-2.15.0/
    It loads the file into memory

   - (evil-mode 1)
    turns Evil mode on globally
    keybindings then switch to Vim-style

   *At this point, Emacs is actively running Evil*

*** After installation

    On the next Emacs launch:
    - Emacs does *NOT* download anything again
    - MELPA is *NOT* contacted
    - _package.el_ is only used for loading already installed packages

    Instead Emacs does:
    - looks in ~/.config/emacs/elpa/
    - finds evil-2.15.0 directory
    - loads it locally when needed
